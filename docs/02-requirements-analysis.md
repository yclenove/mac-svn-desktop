## 1. 分析目的

对 SRS 中的功能需求进行可行性拆解与风险识别，明确每个模块「靠什么 svn 能力实现、数据从哪来、难点在哪」，为概要设计提供依据。

## 2. 核心用例分析

### 2.1 用例总览

```
                          ┌────────────────────┐
     开发者 ─────────────▶│ UC1 日常提交流      │
                          │ UC2 更新与解决冲突   │
                          │ UC3 查看历史与差异   │
     技术负责人 ─────────▶│ UC4 分支管理与合并   │
                          │ UC5 浏览远端仓库     │
     非CLI用户 ──────────▶│ UC6 图形化增删改还原 │
                          └────────────────────┘
```

### 2.2 UC1 日常提交流（P1，使用频率最高）

**主流程：** 打开 WC → 查看变更列表 → 逐个看 diff 确认 → 勾选文件 → 填写中文说明 → 提交 → 得到新 revision。

**分析要点：**

- 频率最高的路径，性能与稳定性优先级最高；status 与 diff 必须异步且可取消；
- 中文说明乱码是 CLI 用户最常见痛点，根因是终端 locale 与 svn `--encoding` 不匹配。应用内统一以 UTF-8 编码写入、显式传 `--encoding UTF-8`，可根治；
- 提交部分文件（changelist 语义）用「勾选文件路径列表」直接传参实现，不引入 svn changelist 概念，降低理解成本。

**异常分支：**

| 异常 | 应对 |
|------|------|
| 提交时远端已有新版本（out of date） | 识别 E155011/E170004 类错误，提示先 Update |
| 选中文件含冲突 | 提交前置检查阻止（FR-CM-04） |
| 认证失败 | 弹凭据框重试（FR-AU-01） |
| 网络中断 | 展示 svn stderr 摘要 + 「检查 VPN/网络」建议 |

### 2.3 UC2 更新与解决冲突（P1 触发 + P3 解决）

**主流程：** Update → 冲突计数提示 → 进入冲突列表 → 三路合并编辑器逐块处理 → 标记 resolved → 重新提交。

**分析要点：**

- svn 冲突时在 WC 内落下四个产物：`file.mine`、`file.rBASE`、`file.rHEAD`、含冲突标记的工作文件。`svn info --xml` 的 `<conflict>` 节点给出三方文件准确路径——这是三路合并数据源，**不需要自己解析冲突标记**；
- 交互式冲突提示必须规避：所有 update/merge 统一加 `--accept postpone`，把冲突留给 UI 处理；
- 三路合并引擎（diff3 语义）需要自研：对 Base/Mine/Theirs 做两两 LCS 对齐，归并为「稳定块 + 冲突块」序列。这是全项目算法复杂度最高的部分，必须纯函数化 + 充分单测。

**关键数据流：**

```
svn update --accept postpone
     │ 状态含 C
     ▼
svn status --xml ──▶ 冲突文件列表
     │
svn info --xml <file> ──▶ base/mine/theirs 三方文件路径
     │
MergeEngine.merge(base, mine, theirs) ──▶ [稳定块|冲突块] 序列
     │ 用户逐块决策
     ▼
写回工作文件 ──▶ svn resolve --accept working <file>
```

### 2.4 UC3 查看历史与差异（P1/P2）

- `svn log --xml -v` 提供结构化输出（含变更路径列表），分页用 `-l 100` + 下一批以上一批最老 revision-1 为起点；
- 任意两版本文件 diff：`svn diff -r N:M <target>` 或分别 `svn cat -r` 后本地 diff（复用 MergeEngine 的对齐算法做两路 diff，一份代码两处使用）。

### 2.5 UC4 分支管理与合并（P2）

- 分支列表本质是「约定目录的远端 list」：`svn list --xml ^/branches`。目录约定必须可配置（FR-BR-01），因为并非所有仓库遵循标准布局；
- `svn switch` 前必须检查本地未提交变更（status 非空即警告），避免用户切换后混淆变更归属；
- Merge 向导的「预览」用 `svn merge --dry-run` 实现，输出为文本格式（无 `--xml`），需要专门的行解析器——这是少数无法用 XML 的场景，解析器要容错。

### 2.6 UC5 浏览远端仓库（P2）

- `svn list --xml --depth immediates <url>` 懒加载一层，配合 UI 树节点展开事件；
- 大目录（万级条目）需虚拟化列表渲染；
- `svn cat` 预览设 5 MB 上限（NFR 保护），二进制探测（内容含 NUL 字节或 `svn:mime-type` 非 text）后拒绝预览。

### 2.7 UC6 图形化增删改还原（P1）

- revert 是不可逆操作（本地修改丢失），必须二次确认并列出受影响文件；
- `svn delete` 对未提交新增文件（A 状态）与已纳管文件行为不同，前者等价 revert + 保留/删除磁盘文件的选择，UI 需区分文案。

## 3. 技术可行性分析

### 3.1 svn CLI 能力映射表

| 功能 | svn 命令 | 输出格式 | 可行性 |
|------|----------|----------|--------|
| 状态 | `status --xml` | XML | ✅ 稳定 |
| 日志 | `log --xml -v` | XML | ✅ 稳定 |
| 信息/冲突定位 | `info --xml` | XML | ✅ 稳定（1.8+ 含 conflict 节点） |
| 远端列表 | `list --xml` | XML | ✅ 稳定 |
| Blame | `blame --xml` | XML | ✅ 稳定 |
| 属性 | `proplist --xml` / `propget` / `propset` | XML/文本 | ✅ |
| 提交 | `commit --encoding UTF-8` | 文本 | ✅ 只需解析末行 revision |
| 更新 | `update --accept postpone` | 文本 | ⚠️ 逐行解析动作前缀（A/U/D/C/G） |
| 合并 | `merge [--dry-run]` | 文本 | ⚠️ 同上，需容错解析 |
| 差异 | `diff` | unified diff | ✅ 标准格式 |
| 锁定 | `lock` / `unlock` | 文本 | ✅ 简单 |
| 检出 | `checkout --depth X` | 文本（进度行） | ✅ 逐行进度 |

结论：**结构化查询全部有 `--xml` 保障；写操作输出简单可解析。CLI 路线对全部 P1–P4 需求可行。**

### 3.2 认证可行性

- svn CLI 默认将凭据缓存到 macOS Keychain（`store-passwords` 默认开启），**应用首选"透传"策略**：不管理密码，认证失败时才介入；
- 介入方式：`--username X --password-from-stdin --non-interactive`，密码经 stdin 写入，满足 NFR-04；
- svn+ssh 依赖用户 `~/.ssh` 配置，应用只做检测与文案引导，不管理 ssh key（边界明确）。

### 3.3 性能分析

| 场景 | 风险 | 对策 |
|------|------|------|
| 10 万文件 WC status | XML 输出可达数十 MB | 流式 SAX 解析（`XMLParser`），边解析边分批投递 UI |
| 进程启动开销（每次 ~30ms） | 高频小查询累积延迟 | 操作合并（如 status 后不逐文件 info）；结果缓存 + 失效策略 |
| 大文件 diff/合并 | 内存与渲染压力 | 10 MB 以上文件禁用内置视图，提示外部工具 |
| 远端大目录 list | 请求慢 | 懒加载 + 超时可取消 + 缓存 |

### 3.4 三路合并引擎可行性

- 算法基线：Myers diff（O(ND)）做两路对齐 → diff3 归并规则生成块序列；
- 参考语义：GNU diff3 与 git merge-file 的冲突判定（双方对同一区域的不同修改 = 冲突；单方修改 = 自动采纳）；
- 规模控制：按行 diff，行内字符级高亮只在渲染冲突块时二次计算；
- 验证手段：与 `diff3 -m` 输出做对拍测试（macOS 自带 diff3），保证语义一致。

## 4. 风险登记册

| # | 风险 | 概率 | 影响 | 缓解措施 |
|---|------|------|------|----------|
| R1 | 三路合并引擎语义错误导致合错代码 | 中 | **高** | 与 diff3 对拍；冲突块保守判定（宁可多报冲突）；结果区允许手动编辑兜底 |
| R2 | svn 文本输出（update/merge）格式随版本/语言变化 | 低 | 中 | 强制 `LC_ALL=C`；解析器容错（无法识别的行忽略并记日志） |
| R3 | 大 WC 性能不达 NFR-01 | 中 | 中 | 流式解析 + 虚拟化列表；性能测试用脚本生成 10 万文件仓库验证 |
| R4 | 用户本机无 svn / 版本过低 | 高 | 低 | 首启检测 + 安装引导页（FR-SE-02） |
| R5 | svn+ssh 环境差异导致认证失败难排查 | 中 | 低 | 检测 + 文档引导，明确不管理 ssh 配置 |
| R6 | 树冲突场景繁多（本地改/远端删等 9 种组合） | 中 | 中 | P3 只支持「保留本地/接受远端」两种粗粒度决策，复杂场景引导 CLI |
| R7 | 签名公证流程（P4）依赖付费开发者账号 | 低 | 低 | P4 前确认账号；期间以未签名构建 + 本地绕过指引分发 |

## 5. 依赖与假设

1. 用户本机可安装 Homebrew 与 Subversion（公司内网机器已验证 svn 1.14.5）；
2. 仓库服务端为 SVN 1.8+（`info --xml` conflict 节点、mergeinfo 依赖）；
3. 分支布局默认 trunk/branches/tags，非标准布局通过设置覆盖；
4. 首版仅中文 UI 不阻塞任何目标用户。
